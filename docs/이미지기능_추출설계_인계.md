# 이미지 기능 — 캐릭터/관계 추출 설계 인계 문서

다른 AI/개발자가 이어서 작업하기 위한 컨텍스트. (작성: 번역 파이프라인 리팩터링 완료 시점)

---

## 0. 프로젝트 한 줄 요약

w.LiGHTER = AI 웹소설 글로벌 현지화 어시스턴트. 한국어 웹소설을 4개국(일/미/중/태)으로
번역·현지화하고, 표지/관계도 이미지와 현지화 가이드를 생성하는 서비스.
지금까지 "번역" 기능(app/translation)은 qdrant RAG 연동·리팩터링 완료. 이제 "이미지" 기능 차례.

## 1. 현재 폴더 구조 (중요)

```
app/
└── translation/          ← 번역 모델링 엔진 (완성, qdrant RAG 기반)
    ├── translation_pipeline.py  (TranslationPipeline: 검색→번역→검수→inspect)
    ├── config.py
    ├── retrieval/ (idiom/annotation retriever + 공통 toolbase)
    ├── agents/ (translator, reviewer, inspector, chatbot)
    ├── text_processing/, core/
backend/services/         ← app/ 모델링을 호출해 서빙(HTTP)
    ├── translation_service.py  (app.translation 사용)
    ├── image_service.py        ← 현재 이미지 (보강 필요, 아래 참고)
    ├── guide_service.py, cover_plan_service.py
api_server.py             ← HTTP 엔드포인트
qdrant_local/             ← 로컬 벡터DB (idiom_jp/us/cn/th, kculture)
data/                     ← RAG 데이터
```

**설계 원칙**: app/ = "모델링(LLM/AI 로직)", backend/services = "그걸 호출해 서빙".
번역처럼, 이미지도 모델링 부분을 **app/image/** 로 새로 만드는 방향.

## 2. 목표 — 이미지 기능에 빠진 것

요구사항(테스트 계획 docx)의 이미지 시나리오:
- **TST-IMG-001 표지**: "원문에서 추출된 등장인물 정보 + 사용자 추가 문구"로 gpt-image-2 생성.
- **TST-IMG-002 관계도**: "원문에서 추출된 관계 정보"로 생성.
- 공통: "AI 생성 이미지입니다" 안내 문구, 응답 90초 이내.
- 예외(표지): 나체 등 부적절 요청 시 **단순 거부가 아니라 "거부 + 대안 제시"**.
  예) "성기 노출은 불가하나, '상의 탈의 + 젖은 잠방이로 최소한 가린 모습'으로 조정하면 생성 가능".

### 핵심 갭
현재 `backend/services/image_service.py`는 **캐릭터/관계 정보를 사용자가 payload로 직접
넘겨준다고 가정**한다 (payload.get("protagonist"), payload.get("characters") 등).
그러나 요구사항은 **"소설 원문 → 등장인물·관계 자동 추출(LLM)"** 단계를 요구한다.
→ 이 "추출" 단계가 통째로 없음. **이것을 새로 구현하는 것이 이번 작업의 핵심.**

## 3. 현재 image_service.py 상태 (재사용 가능 부분)

- `cover_image(payload)`: 캐릭터 정보(이미 정리됨) 받아 표지 프롬프트 조립 → gpt-image. (②생성 절반)
- `relation_image(payload)`: characters/relations 받아 관계도 생성.
- `_unsafe_visual_request(text)`: 나체 등 부적절 키워드 차단 (단순 거부만 — 대안 제시 없음).
- `_visual_refusal()`: 거부 응답 (대안 제시 보강 필요).
- `generate_image(prompt)`: gpt-image-2 실제 호출. mock 모드 지원.
- 모델: 환경변수 OPENAI_IMAGE_MODEL, 기본 "gpt-image-2".

## 4. 구현해야 할 것 (단계)

```
[원문] → ① 추출(LLM): 등장인물 정보 + 관계 추출   ← 새로 만들 핵심 (app/image/)
       → ② 생성: 추출결과 + 추가문구 → gpt-image   ← 기존 image_service 절반 활용/이전
       → 안전검사: 부적절 요청 거부 + 대안 제시      ← 기존 거부 로직 보강
```

### ① 추출 (가장 먼저, app/image/character_extractor.py 신규)
- 입력: 소설 원문 (1화 단위로 시작. 관계는 여러 화 볼수록 풍부 — 범위는 결정 필요).
- 출력 형식 (docx 예시 기준):
  - 캐릭터: 이름, 성별, 나이(추정), 외형(리스트), (선택)성격
  - 관계: 인물A →(관계: 설렘/적대 등)→ 인물B
- 방식: LLM(gpt-4.1-mini 등) 호출. **추출은 RAG 불필요할 가능성 높음**(원문만 보면 됨).
  번역과 달리 외부 지식 검색이 필요 없는 순수 추출 작업.
- 출력은 구조화(JSON/dataclass) → ②생성이 그대로 받아쓰게.

### ② 생성 (app/image/image_generator.py — 기존 image_service 로직 이전/정리)
- 추출 결과(캐릭터/관계) + 사용자 추가 문구 → 프롬프트 조립 → gpt-image-2.
- 표지(인물 중심) / 관계도(관계 맵) 두 종류.

### 안전검사 (거부 + 대안)
- 기존 _unsafe_visual_request 활용하되, _visual_refusal을 "대안 제시 포함"으로 보강.
- docx 예외 케이스 참고: 부적절 부분만 짚고 "이렇게 조정하면 가능"을 LLM이 생성하도록.

## 5. 설계 시 먼저 정할 결정들 (미해결)

1. **추출 입력 단위**: 1화? 여러 화? (관계 추출은 누적이 유리)
2. **추출 출력 형식**: 캐릭터당 필드 확정 (이름/성별/나이/외형/성격?), 관계 표현 형식.
3. **표지 인물 수**: 주인공 1명? 다수?
4. **추출에 RAG 쓸지**: 권장은 순수 LLM(원문만). 단 한국 문화 맥락이 필요하면 kculture RAG 고려.
5. **app/image 와 backend/services/image_service 의 역할 분담**:
   app/image = 추출+생성 모델링, image_service = 그걸 호출+HTTP 포장.

## 6. 참고 — 번역 기능이 쓴 패턴 (이미지도 일관되게)

- config로 모델명·파라미터 관리 (예: app/translation/config.py).
- mock 모드(WLIGHTER_MOCK_MODE=true)면 가짜 응답, false면 실제 LLM. 기본 false.
- 검증 스크립트: tests/check_translation_pipeline.py 처럼 tests/check_image_*.py 만들어
  mock=False로 "원문→추출→생성" end-to-end 확인 + HTML 출력 권장.
- 결과를 LLM/사용자에게 줄 때 노이즈(내부 ID, 점수) 빼고 필요한 필드만.

## 7. 검증 방법

- 단위 테스트: 추출 결과 형식, 안전검사 동작 (mock).
- end-to-end: 실제 원문(tests/엘리트_1화.txt 같은 한국 웹소설 원문)으로 추출→생성 확인.
- docx 테스트 케이스(TST-IMG-001/002) 입력으로 기대 출력 비교.
- 응답 시간 90초 이내 측정.
